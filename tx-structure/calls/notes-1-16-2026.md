## Motivation and Problem Statement

If a Payjoin (PJ) transaction uses inputs drawn from two strong clusters, that provides strong evidence the transaction is collaborative.

Key questions:

1. Can we identify a transaction as a Payjoin from surrounding graph context?
2. Can we build better clustering heuristics than MIH by incorporating this context?

A core issue is that wallet fingerprints leak heavily through transaction structure. Even if a single transaction looks ambiguous in isolation, its surrounding graph often is not.

## Fingerprinting and Wallet Behavior

One possible mitigation is encouraging wallets to converge on similar fingerprints:

* Examine transaction neighborhoods up to some depth.
* Use empirical evidence from surrounding transactions to adapt wallet behavior.
* Effectively, wallets “learn” a fingerprint distribution rather than optimize in isolation.

However, there is likely no perfect solution here, which motivates our focus on multi-party Payjoins (MP-PJs).

We should:

1. Define the privacy goal first (indistinguishability, k-anonymity, etc.).
2. Design protocols that approximate it at the sub-graph level, not per-transaction.

## Graph-Level Perspective

Due to the transaction graph structure:

* Each coin implicitly carries the history of wallet fingerprints that produced it.
* A single transaction cannot reset this history.

Achieving meaningful k-anonymity requires:

* High connectivity within a subgraph.
* A lower bound on entropy within that subgraph.

Without sufficient connectivity, anonymity collapses once coins are spent again.

## Why Multi-Party Payjoins ?

If we ship a library that allows normal wallet transactions to participate in MP-PJs:

* Even a subset of users constructing ambiguous values and dense subsets benefits others.
* Users not explicitly optimizing for privacy still gain plausible deniability.

MP-PJs may still be fingerprintable, but:

* The adversary can at best infer this user optimizes for privacy.
* They cannot reliably extract clustering information.

This contrasts with CoinJoin systems that rely on isolated per-transaction k-anonymity, which degrades rapidly on subsequent spends.

Main point: Fixing fingerprinting within a single transaction does not fix privacy at the transaction-graph level.

## Theory Gaps

* Existing theory for Payjoin privacy within the sub-transaction model is limited (e.g., UIH-style arguments).
* Wallet fingerprints are a dominant signal.
* Signals include:

  * Output amount Hamming weight
  * Structural fingerprints

A separate-entity interpretation often becomes the most plausible explanation, reducing entropy.

## Core Insight

Bitcoin privacy fundamentally depends on others' use of Bitcoin.

We cannot provide strong theoretical guarantees over:

* The global transaction graph, or
* Arbitrary subgraphs.

Privacy is an emergent property of collective behavior, not individual optimization.

## Technical Considerations

### Edge and Vertex Removal

* If a user fully exits a subgraph (liquidates), and the subgraph is a superconcentrator, only *their* privacy is harmed.
* If a user publishes their BIP32 master key:

  * Remove their edges from the graph.
  * Set their subjective probability weights to zero.
  * Measure remaining entropy over associated coins.

Percolation theory suggests a **critical threshold** where the graph remains maximally connected and worst-case attacks are minimized.

## Graph Construction Constraints

Efficient construction must account for:

* Transactions with hundreds of edges.
* Dense subsets within transactions.
* Low Hamming-weight output values as a structural feature.
